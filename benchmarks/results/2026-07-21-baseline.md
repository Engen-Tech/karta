## Read this before reading the verdict

**`fail=0` does not mean karta is clean.** The probes deliberately separate *gating* from *evidence*: a
probe fails only on its own integrity (fixture-detection matrices, manifest gates, missing enrolled repos),
while field findings are recorded and never flip the status. That stance is documented per probe — for
example field-delivery-state-audit: *"gating = fixture, live = recorded evidence."*

So the honest headline is two numbers, not one:

- **0 vectors failed** — every implemented probe's own integrity held.
- **103 findings recorded** across the 13 measured vectors, including 14 doc-truth violations, one high-severity
  unescaped-input surface, and 3 mirror-sync integrity errors.

And **11 of 24 vectors are unmeasured, not passing.** The runner prints this on every run and it is not
softened here.

## What the findings actually say

**Documentation is the largest measured defect surface (14 violations).** README claims five skills against a
truth of 10 and two agents against a truth of 4; `docs/how-to/claude-code.md` claims three agents; the marketplace
manifest claims two read-only gate agents; `docs/how-to/codex.md` still pins "Karta 1.19" at current major 2;
`karta-status` appears in no user-facing doc; and the label-root table omits `docgardner` and `kaizen`. These are
mechanically checked counts, so they are fixable and re-verifiable.

**One high-severity input surface.** `serve_status.py` renders attacker-controlled binder strings with zero
`html.escape` calls — 7 unescaped sinks, 3 reaching rendered HTML. The frozen sink list is recorded so the count
must reach and stay 0. Separately (medium), `inject_karta_status.py` carries repo-derived text into session
context with no inert delimiter, so a poisoned binder string arrives as undelimited instruction.

**Three mirror-sync integrity errors.** Write-mode sync compares bytes only, so the exec bit never reaches
`.agents/skills/` or `plugins/karta/`; a degraded lock entry makes sync rmtree an external skill as an orphan
then crash on the stale have-set; and a tampered external `SKILL.md` passes both `sync --check` and
`validate_plugin` because `computedHash` is never recomputed against content.

**Verify-loop caps are in-memory.** Both gate agents declare "store no loop state" with no persisted-attempt
mechanism, so attempt caps reset on resume; the safety auditor's BLOCKED definition also lacks the empty-diff
clause, so a readable-but-empty diff fires no signal.

**Cost is growing.** `bench_loc_delta` +1298 and `script_count_delta` +3 since the last ledger row — recorded
as evidence for the human, never a gate failure.

## Caught during this run: the bench scored itself

The first pass of this baseline showed the `DG-SCHEMA` doc-truth violation resolving — count 2 to 1, apparently
progress. It was not. `DG-SCHEMA` greps `*.py` for `doc-gardner-schema.json` to ask whether karta ships code
backing a documented schema gate, and the lane ran repo-wide with no path exclusion. The bench build-out had
just added `benchmarks/flow/check_contradictions.py`, which names that filename in its own logic. The grep
found it and declared karta's promise met.

No karta code implements the gate. The doc's promise is still unmet; the bench had simply started measuring
itself. The sibling ledger entry `KAIZEN-USERREPO` already carried `"scopes": ["scripts", "hooks", "skills"]`
— `DG-SCHEMA` was the one entry that omitted it, so the fix was data, not code, and it restores the violation
rather than silencing one (a tightening, consistent with the add-only rule).

Two things worth carrying forward:

1. **This is the failure mode `bench-self-integrity` exists to catch, and that vector is one of the 11 with no
   probe.** The bench found this defect by accident, on a run whose purpose was something else. Any lane that
   asks "does the repo contain code doing X" is contaminable the moment the bench itself writes about X.
2. **A shrinking violation count is not self-evidently good news.** The comparison rule already says the metric
   is the per-lane vector and that silencing must show in the diff; this run is the case that proves why. Every
   future improvement in these lanes should be confirmed against the actual repo before it is recorded as one.

## Run reproduction

```
python3 benchmarks/gate/run_gate.py --date 2026-07-21 \
  --consumers ../parchmark,../gringotts
```

`--consumers` is optional in form but not in practice: without it, probes guess the consumer repos as
siblings of this checkout's parent, and any run from a worktree fails three probes closed on repos that
exist elsewhere. This baseline was produced with the paths stated explicitly, which is why the same 13/13
result now reproduces from any checkout location.
